### 第二次进入深度睡眠+ULP程序处理无法退出了

## 问题描述

第一次进入深度睡眠+ULP程序处理正常，可以正确唤醒主CPU。但是第二次进入深度睡眠+ULP程序处理时，ULP程序卡在了`sensor_init()`函数中，无法继续执行，导致无法唤醒主CPU。

## 根本原因分析

### 1. 问题定位

第二次无法唤醒主CPU是因为ULP程序卡在了`sensors_init()`函数中。第一次进入的时候，正确关闭了主CPU的I2C等外设，然后给ULP配置RTC的I2C外设。但是唤醒主CPU的时候，ULP我并没有"删除"这个ULP程序，我为什么把删除加双引号（因为在 ESP32-S3 中，ULP RISC-V 没有专门的"删除"指令，所谓的"删除并重新初始化"实际上是指重置内核状态、清除定时器、清空 RTC 内存并重新加载固件。）需要这样:

```
if(isNotFirst)
    ulp_riscv_halt(),ulp_riscv_reset(); // 硬件复位 ULP，确保寄存器和状态完全清零，避免残留状态干扰下一轮监测。
```

然后我找解决方案的时候，一直问AI，但是AI一直在找I2C总线的问题，说MAX30102因为总线切换卡死了，但怎么可能卡死？第一次进入的时候两个传感器都没有问题，怎么可能第二次就卡死了。第一次成功，第二次失败，说明硬件环境没变，变的是 ULP 的内部状态。

### 2. ESP32-S3 中 ULP 的运行机制

在 ESP32-S3 中，主 CPU 唤醒后，ULP 内核并不会自动"消失"。如果主 CPU 只是简单地重新调用 load_binary 和 run，确实会产生以下问题：

- 内核并未真正复位：ULP RISC-V 的程序计数器（PC）和寄存器状态可能残留，甚至之前的代码逻辑还在某个死循环里跑。
- RTC I2C 硬件状态机冲突：ULP 的 I2C 是硬件实现的。如果第一次 ULP 程序在 i2c_read/write 过程中被强行中断（或者虽然执行了 halt 但硬件状态机没复位），下一次启动时，硬件 I2C 模块可能认为自己还在上一次的事务中，导致 sensors_init 无法发起新的 Start 信号。

## 解决方案

在主 CPU 的 `init_ulp_program()` 函数中，在第一次运行之后，需要调用 `ulp_riscv_halt()` 和 `ulp_riscv_reset()` 来彻底复位ULP内核：

```c
static void init_ulp_program(void) {
    static bool isNotFirst = false;
    ulp_timer_stop();

    if (isNotFirst) {
        // 第一次运行后，必须彻底复位ULP
        ulp_riscv_halt();
        ulp_riscv_reset(); // 硬件复位 ULP，确保寄存器和状态完全清零
    }

    esp_err_t err = ulp_riscv_load_binary(ulp_main_bin_start, (ulp_main_bin_end - ulp_main_bin_start));
    ESP_ERROR_CHECK(err);

    // ... 其他初始化代码 ...

    /* Start the program */
    err = ulp_riscv_run();
    ESP_ERROR_CHECK(err);

    isNotFirst = true;
}
```

## 关键点

1. **ULP 的状态残留问题**：ULP RISC-V 内核不会自动清零，需要显式调用 `ulp_riscv_reset()` 硬件复位
2. **RTC I2C 硬件状态机冲突**：ULP 使用硬件 I2C，如果上一次的事务没有正常完成，状态机会卡住
3. **不是 I2C 总线切换问题**：硬件环境没变，变的是 ULP 的内部状态
4. **硬件复位的重要性**：`ulp_riscv_halt()` + `ulp_riscv_reset()` 确保完全清除内核状态和 RTC I2C 硬件状态机

## 经验教训

- 在 ESP32-S3 中，`ulp_riscv_halt()` 只是停止执行，不会清零内核状态
- 每次重新加载 ULP 程序前，必须调用 `ulp_riscv_reset()` 进行硬件复位
- 硬件 I2C 模块的状态机需要在每次 ULP 运行前彻底复位
- 不能假设第二次运行时内核状态会和第一次一样干净
